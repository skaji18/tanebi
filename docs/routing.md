# タスクルーティング設計

リクエストを受け取ったとき、オーケストレーター（Core のフロー決定ロジック）は **TANEBIフロー（DECOMPOSE→EXECUTE→AGGREGATE→LEARN）** を起動するか、**直接応答** するかを判断する必要がある。

本ドキュメントはその判定基準を定義する。フロントエンド実装（Claude Code / Listener / カスタムUI 等）に依存しない、フレームワーク共通の設計である。

---

## 設計背景

TANEBIフローのオーバーヘッド:

| ステップ | コスト |
|---------|-------|
| DECOMPOSE | Decomposer 1回の実行コスト |
| CHECKPOINT | Checkpoint Worker 1回の実行コスト |
| AGGREGATE | Aggregator 1回の実行コスト |
| イベントログ | 軽微（YAMLファイル数十件） |

TANEBIフローの価値:

| 価値 | 条件 |
|------|------|
| 並列化 | 独立サブタスクが複数存在する |
| 品質検証 | Checkpoint によるフィードバックループ |
| 知識蓄積 | Learning Engine へのシグナル供給 |
| コンテキスト管理 | Worker ごとに独立したコンテキスト |
| 監査証跡 | イベントログによる再現性 |

**損益分岐点**: 独立サブタスクが3つ以上あれば、並列化のゲインがオーバーヘッドを上回る。

---

## 判定基準：3独立サブタスク閾値

リクエスト受信時に以下の判定を行う:

> **「このリクエストを独立して並列実行できるサブタスクに分けたとき、3つ以上になるか？」**

この問いへの回答でリクエストを3カテゴリに分類する。

### カテゴリ A: 直接応答（TANEBIフロー不使用）

独立サブタスクが **0〜1個** のリクエスト:

- **質問・相談**: 仕組みの説明、選択肢の比較、エラー原因の調査
- **状態確認**: タスク状態の参照、ログの確認
- **単一ファイルの変更**: 設定値の変更、誤字修正
- **フレームワーク自体の操作**: tanebi CLI の実行、work/ の確認
- **既存タスク結果への追加質問**

### カテゴリ B: TANEBIフロー（完全実行）

独立サブタスクが **3個以上** のリクエスト:

- **3つ以上の独立サブタスク** がリクエスト文から即座に想定できる
- **複数ドメイン** にまたがる作業（例: backend + frontend + test + docs）
- **品質検証が重要** で、Checkpoint によるフィードバックループが価値を持つ
- **反復的なタスクパターン** で、Learning Engine のシグナル蓄積が将来の品質向上に寄与する

→ タスク作成（`tanebi new`）から開始し、フロー全体を省略なく実行する。

### カテゴリ C: グレーゾーン（2サブタスク前後）

独立サブタスクが2個前後で判断が割れる場合、以下のタイブレーカーを **上から順に** 適用する:

| 優先順 | 条件 | 判定 |
|--------|------|------|
| 1 | `knowledge/signals/` に該当ドメインのシグナルが既に存在する | → TANEBIフロー（学習蓄積の継続価値） |
| 2 | 品質検証が重要なリクエスト（本番コード変更、API設計等） | → TANEBIフロー（Checkpoint の価値） |
| 3 | リクエストに品質期待が明示されている | → TANEBIフロー |
| 4 | 上記いずれにも該当しない | → 直接応答 |

---

## ルーティング宣言

オーケストレーターは判定結果をユーザーに伝える。カテゴリ A/B（明確な場合）は結論のみ、カテゴリ C（グレーゾーン）は理由を付記する。

宣言の形式はフロントエンド実装に委ねる（テキスト表示、UIバッジ、ログ記録等）。

---

## エスカレーション：直接応答 → TANEBIフローへの昇格

直接応答を選択した後、作業中に以下を検知した場合は **途中からTANEBIフローに切り替える**:

- 当初の見積もりより変更範囲が広い（3ファイル以上に波及）
- 独立して作業可能な部分が3つ以上あることが判明
- 品質検証が必要な複雑さに達した

エスカレーション手順:
1. それまでの調査結果をリクエスト内容に含めて `tanebi new` を実行
2. 通常のTANEBIフロー（DECOMPOSE 以降）を開始
3. 調査済みの情報を request.md に記載することで、Decomposer の再調査を防ぐ

---

## 禁止事項

- **全リクエストをTANEBIフローに通す**: 質問や単一ファイル修正へのTANEBIフローはオーバーヘッドの浪費
- **全リクエストを直接応答にする**: 複雑タスクの直接処理はコンテキスト管理の破綻と品質低下を招く
- **ルーティング宣言の省略**: ユーザーはどちらのパスが選ばれたか知る権利がある
- **グレーゾーンで安易に直接応答を選ぶ**: タイブレーカーを明示的に適用すること
